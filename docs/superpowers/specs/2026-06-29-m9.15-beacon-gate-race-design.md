# M9.15 — Beacon-connectivity gate race: deterministic offline reproduction + fix

**Date:** 2026-06-29
**Milestone:** M9.15 (live two-binary `mixed_network`), rung-3 follower bootstrap
**Status:** design — approved for planning

## Problem

M9 is complete offline; the only remaining work is the nightly-gated live
two-binary `mixed_network` arm (a real Go 5-validator cluster + a Rust
follower converging to the same tip). That arm is blocked at one rung: the
Rust follower's **production beacon-connectivity gate is racy**. The follower
TLS-handshakes all 5 Go validators (Go ground-truth confirms the peer is
connected and receiving `GetPeerList` pulls), but the bootstrapper never emits
`GetAcceptedFrontier`, so it stalls — only ~2 of 6 live runs reach NormalOp.

Diagnosis from live runs is blocked by a *meta-blocker*: the follower's
`tracing` subscriber goes silent at ~14 ms under the harness's
concurrent-connection load, hiding the decisive rung-4/5/7 markers. This design
**sidesteps** that meta-blocker by reproducing the race deterministically
in-process, rather than relying on live logs.

### Root cause framing

The production connectivity-gate chain:

```
peer TLS handshake completes
  → Peer::finish_handshake()           [synchronous, per-peer, on N concurrent tasks]
    → router.connected(node_id, version, PRIMARY)
      → InsecureValidatorManager       [optional, only when sybil protection disabled]
        → BeaconManager.connected():
             if subnet == PRIMARY && beacons.get_weight(node) != 0:
                 num_conns += 1                                   (AtomicI64)
                 if num_conns >= required_conns:  watch.send(true)   ← THE GATE
        → inner ExternalHandler (RouterBridge → engine)
  ... gate fires true ...
    → boot_chain_with_sender start-task: gate.wait_for(true)
      → handler.start() → Bootstrapper broadcasts GetAcceptedFrontier to beacon set
```

Reference: `crates/ava-network/src/peer/handshake.rs:298` (`finish_handshake`),
`crates/ava-node/src/init/networking.rs:182` (`BeaconManager`),
`crates/avalanchers/src/wiring/chains.rs:1340` (gate → `handler.start()`).

**Invariant under test:** for `N ≥ 4` beacons, regardless of the
order/timing/concurrency of `connected()`/`disconnected()` callbacks, the gate
resolves `true` exactly once and `handler.start()` then broadcasts
`GetAcceptedFrontier` to a non-empty beacon set.

**Coverage gap that let the bug escape:** the only existing offline gate test,
`crates/avalanchers/tests/networked_bootstrap.rs`, *hand-fires*
`connected_tx.send(true)` (lines ~220, ~260) after manually checking
`connected_peers()`. The real `BeaconManager` decision logic has **never** run
in an offline test. Single-beacon live works (`required_conns == 1` fires on the
first callback); the ≥4-beacon path is the racy, untested one.

### The three hypotheses (each a perturbation of the chain above)

1. **Undercount** — a `connected()` for a real beacon never reaches
   `BeaconManager` (dropped/filtered) under concurrent load → `num_conns` stalls
   below `required_conns`.
2. **Empty broadcast** — gate fires, but the bootstrapper's beacon set is
   empty/mis-addressed when `start()` runs → `GetAcceptedFrontier` goes nowhere.
3. **Flap decrement** — connect→disconnect→reconnect drives `num_conns` below
   threshold and, because *only* `connected()` re-checks the threshold, the gate
   can wedge.

The prime suspect: the `AtomicI64` counter has **no set-based dedup and no
re-evaluation outside `connected()`**, unlike Go's `node/beacon_manager.go`,
whose `numConns` is backed by a deduplicated peer set with paired
connect/disconnect events.

## Chosen approach — C (both A and B, sequenced)

A deterministically pins and fixes the bug in a fast PR-gated test; B converts
the one-off live flake into a standing end-to-end offline guarantee and closes
the coverage gap.

### Approach A — deterministic router-chain test (the pin)

**New file:** `crates/ava-node/tests/beacon_gate_race.rs` (co-located with
`BeaconManager`; `ava-node` already has `BeaconManager`, `ValidatorManager`,
`watch`, and the router traits in scope). No sockets, no TLS.

**Harness components — all real production types:**
- A real beacon `ValidatorManager` seeded with `N = 5` node ids, weight 1 each
  → `required_conns = (3·5 + 3) / 4 = 4`.
- A real `BeaconManager::new(inner, beacons, required_conns, watch_tx)`, where
  `inner` is a tiny recording `ExternalHandler` that logs every
  `connected`/`disconnected` it receives (asserts pass-through).
- A driver that mimics `finish_handshake`: calls
  `bm.connected(id, &version, Id::default())` — the exact production call site,
  including `Id::default()` as the subnet.

**Adversarial cases (table-driven), each asserting the invariant:**
1. *Happy concurrent burst* — 5 `connected()` calls across tokio tasks; gate
   resolves `true` exactly once, recorded count == 5, inner saw all 5.
2. *Exactly-threshold* — 4 of 5 connect → gate fires; 3 of 5 → gate stays
   `false`.
3. *Flap* — connect 4 (gate fires) → disconnect 1 (count → 3) → reconnect
   (count → 4); assert gate behavior and pin whether a late re-fire is needed
   (hypothesis 3).
4. *Disconnect-before-connect / non-beacon connect* — a `disconnected()` for an
   uncounted node (does `num_conns` go negative and wedge the threshold?); a
   `connected()` for a weight-0 node (must NOT count).
5. *Duplicate connected* — same beacon `connected()` twice with no intervening
   disconnect. **Treated as a must-fix invariant:** the counter must match Go's
   set semantics and not double-count (else it could fire early with < 4
   *distinct* peers).

Cases 3–5 are where the real bug is expected to surface. The test makes
whichever case fires a deterministic red.

### Approach B — multi-beacon TLS regression gate (end-to-end)

**New file:** `crates/avalanchers/tests/beaconed_bootstrap.rs`, generalizing
`networked_bootstrap.rs` from 2 → `N + 1` nodes:
- `N = 5` beacon `Node`s (matching the live `mixed_network` arm's 5-validator
  cluster → `required_conns = 4`; each a beaconless `TestVm` at `TIP_HEIGHT` so
  their Getters answer) + 1 follower.
- The follower's gate is the **real `BeaconManager`** wired into its router
  chain (the production construction from `init_networking`, factored so the
  test can call it) — *not* a hand-fired `connected_tx.send(true)`. This is the
  one change that gives the production gate its first end-to-end offline
  exercise.
- Follower `manually_track`s all 5 beacons; assert it reaches
  `EngineState::NormalOp` and converges on the beacon tip.
- Pre-fix this reproduces the live flake; post-fix it is the standing
  regression gate and the direct offline analog of the nightly live
  `mixed_network` arm.

To keep B deterministic rather than flaky, the follower waits on real
connectivity with a generous bounded timeout (mirroring `wait_until_timeout`),
and the convergence assertion runs only after all 5 handshakes are observed.
The Section-on-fix change is what removes the timing dependence; B confirms it.

### The fix (shape; exact form gated on what A reveals)

Lands in `BeaconManager` (`crates/ava-node/src/init/networking.rs`).
Anticipated shape, matching Go `node/beacon_manager.go` set semantics:
- Replace the bare `AtomicI64` with a **set of connected beacon node-ids**
  (behind a `Mutex`/`parking_lot::Mutex`), so `connected` is idempotent (dedup)
  and `disconnected` removes-by-id; `num_conns` becomes `set.len()`. Structurally
  kills hypotheses 1, 3, 5.
- Evaluate the threshold on **both** `connected` and `disconnected` edges
  (re-fire-safe; the watch already de-dups `true`), so the gate cannot wedge.
- If A instead points at hypothesis 2 (empty broadcast), the fix moves to the
  gate → `handler.start()` beacon-set plumbing in `boot_chain_with_sender`
  (`crates/avalanchers/src/wiring/chains.rs`). The implementation plan will
  branch explicitly once A reports which hypothesis is real.

## Sequencing

1. A (red) → fix → A (green).
2. B (red pre-fix where reproducible / green post-fix) as the regression gate.
3. Confirm the gated live `mixed_network` arm is unblocked (document only — it
   stays `#[cfg(feature="live")] #[ignore]`, nightly-gated; **no live run in
   this work**).
4. Update `plan/M9-interop-hardening.md` with an AS-BUILT note recording the
   pinned hypothesis and the fix.

## Testing & lint

- Standard `./scripts/run_task.sh lint` + `test-unit` (this is
  `ava-node`/`avalanchers`, **not** the stricter `lint-saevm` area).
- A runs every PR (deterministic, fast). B runs every PR (bounded-wait,
  in-process TLS — same class as the existing `networked_bootstrap.rs`).

## Scope & risks

- **Out of scope:** the concurrent-load `tracing`-subscriber death (a separate
  meta-blocker; this design sidesteps it by not relying on live logs); any
  Go-side changes; the actual nightly live run.
- **Risk:** B may stay mildly timing-sensitive even post-fix. Mitigation: keep
  B's assertion behind a bounded wait and treat A as the authoritative
  deterministic pin.
- **Dependency:** factoring the production gate construction out of
  `init_networking` so B can wire the real `BeaconManager` without assembling a
  full node. Keep this a thin, behavior-preserving extraction.
