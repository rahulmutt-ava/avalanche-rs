# M9.15 follow-up — Atomic at-most-once peer dedup in `ava-network`

**Date:** 2026-06-30
**Area:** `crates/ava-network` (`network/net_impl.rs`)
**Milestone:** M9.15 (live mixed-network) documented follow-up
**Spec refs:** `specs/05` §3.1 (peer sets), `specs/17` §2 (network loops), `00` §6.1 (determinism)

## Problem

The network's connect/disconnect notifications are **not at-most-once per peer**.
The inbound accept path does no duplicate-connection check at all, and even the
outbound path's check is not atomic with registration.

Concretely, in `crates/ava-network/src/network/net_impl.rs`:

- `handle_dial` (outbound) checks `connected.contains(node) || connecting.contains(node)`
  *after* the TLS upgrade and *before* spawning a peer actor.
- `handle_accepted` (inbound) does **no such check** — a redundant inbound
  connection from an already-known node spawns a **second** `Peer` actor and a
  second `watch_peer` task.

Consequences of the duplicate inbound actor:

1. `router.connected(node)` fires **again** — it is emitted from inside the peer
   actor's handshake (`peer/handshake.rs:319`), once per actor. → **duplicate Connected.**
2. The `connecting` / `connected` `PeerSet`s are keyed by node-id, so the second
   `insert` silently overwrites the first peer's handle; the two `watch_peer`
   close-paths then each call `router.disconnected(node)`. → **duplicate /
   mispaired Disconnected.**

`BeaconManager` (`crates/ava-node/src/init/networking.rs`) compensates for this
today with a `Mutex<HashSet<NodeId>>` that dedups beacon connections, but every
*other* `ExternalHandler` (the engine router, `RouterBridge`) is left to tolerate
the duplicates on its own. Go avoids the problem at the source: it holds a single
`peersLock` across the compound check against `connectingPeers`/`connectedPeers`
and the insert, for **both** connection directions.

A secondary, narrower defect: because `handle_dial`'s `connecting.insert` happens
later (inside `watch_peer`), *after* its `contains` check, with no lock held
across the two, two near-simultaneous connections (inbound+outbound, or two
inbounds) can both pass the check and both spawn — a time-of-check/time-of-use
(TOCTOU) race that exists on the outbound path too.

## Goal

Make peer admission **atomic and at-most-once on both paths**, so the network
delivers strictly-paired `connected`/`disconnected` notifications to every
`ExternalHandler` — matching Go's single-`peersLock` semantics — and the
downstream handlers no longer need to compensate.

Non-goals: changing the `PeerSet` storage model, touching the handshake protocol,
or removing the `BeaconManager` compensating set (it becomes defense-in-depth).

## Design constraints (why the shape is forced)

1. **`router.connected` fires from the peer actor, not `watch_peer`** — so a
   "spawn-then-drop-the-loser" approach cannot work: any spawned actor will fire
   `connected` once its handshake completes. The duplicate must be prevented
   *before* `Peer::spawn`.
2. **`Peer::spawn` is synchronous** (returns a handle, spawns its tasks
   internally; no `.await`). So `check → spawn → register` can all live inside a
   single non-`async` critical section guarded by one lock.
3. **Both upgraders yield the same concrete IO type** (`TlsStream<TcpStream>`),
   so a single admission method handles both directions; making it generic over
   `IO` additionally lets tests drive it over an in-process `DuplexStream`.

## Components

### 1. `NetworkImpl.peers_lock: parking_lot::Mutex<()>`
A single **leaf lock** serializing every compound membership transition across
the two existing `PeerSet`s (`connecting`, `connected`). It is the Rust analogue
of Go's `peersLock`.

- Held **only across synchronous sections** (membership check, `Peer::spawn`, set
  inserts/removes) — **never across an `.await`**.
- **Never nested** with `tracked_ips`. Where `watch_peer` currently calls
  `record_success` (which locks `tracked_ips`), that call stays *outside* the
  `peers_lock` section, keeping the two locks disjoint and order-free.

### 2. `fn admit_peer<IO>(&self, node_id, cert, direction, tls: IO) -> bool`
Generic over `IO` (both call sites instantiate at `TlsStream<TcpStream>`; tests
pass `DuplexStream`). Body:

```text
let _g = self.peers_lock.lock();                 // sync critical section
if self.connected.contains(&node_id) || self.connecting.contains(&node_id) {
    tracing::debug!(%node_id, "duplicate connection rejected");   // Go parity log
    return false;                                // `tls` dropped → conn closes; NO actor spawned
}
let handle = Peer::spawn(/* peer_config, node_id, cert, direction, tls, net_token, tasks */);
self.connecting.insert(handle.clone());
drop(_g);
self.spawn_watcher(handle);                      // the former tail of watch_peer
true
```

Returns `true` if this connection was admitted, `false` if it lost the race /
was a duplicate (the caller simply returns; dropping `tls` closes the socket).

### 3. `watch_peer` → `spawn_watcher` refactor
The existing `watch_peer` is split: the leading `self.connecting.insert(handle)`
moves into `admit_peer` (done atomically under `peers_lock`), and the remaining
`select!` watcher loop becomes `spawn_watcher`. Inside the watcher:

- the **promote** transition (`connecting.remove` + `connected.insert` on
  handshake completion) takes `peers_lock`;
- the **close** removals (`connecting.remove` + `connected.remove`) take
  `peers_lock`;

so every membership transition is atomic with respect to admission. `record_success`
and the metrics calls (`observe_connected` / `observe_disconnected`) remain where
they are, outside the `peers_lock` section.

### 4. Call sites
`handle_accepted` and `handle_dial` both call `admit_peer(...)` after their
respective `upgrade()` succeeds:

- `handle_dial` loses its now-subsumed inline `connected.contains || connecting.contains`
  check.
- `handle_accepted` gains dedup for the first time; the stale
  "M9.15 follow-up: unlike `handle_dial`, the inbound path does not dedup…"
  comment is removed (resolved).

## Testing

### T1 — RED unit test (deterministic regression guard)
A `#[cfg(test)] mod tests` block in `net_impl.rs` (so it reaches the private
`admit_peer` and the private `connecting`/`connected` sets). Build a
`NetworkImpl`, then call `admit_peer` twice for the **same** node-id over two
in-process `DuplexStream`s:

- assert the first call returns `true` and the second returns `false`;
- assert `connecting.len() == 1` (no second actor admitted).

This pins the atomic gate directly. The handshake need not complete to prove
admission dedup — preventing the second actor is precisely what prevents the
duplicate `router.connected` (which is emitted per-actor).

### T2 — integration end-to-end proof
In `tests/network_dispatch.rs` (real loopback TLS): two `TestNetwork`s
`manually_track` each other and both `dispatch`, producing a racing
inbound+outbound for the same peer on each side. Assert each side's
`RecordingRouter` records the peer **connected exactly once** (wait until
connected, settle briefly, then assert count == 1). End-to-end proof that no
duplicate notification reaches an `ExternalHandler`. The mild timing dependence
is acceptable because T1 is the deterministic guard; T2 is the realistic
cross-check.

### Unchanged
The 4 `ava-node` `BeaconManager` tests
(`duplicate_connected_does_not_double_count`,
`disconnect_before_connect_does_not_wedge_gate`, `concurrent_connects_fire_gate`,
`beacon_manager_fires_gate_at_required_conns`) stay green — the manager's set is
now defense-in-depth, and its doc comment is updated to note that `ava-network`
guarantees at-most-once on both paths.

## Error handling / edge cases

- **Loser connection:** dropping `tls` on the `false` return closes the TCP/TLS
  connection (Go closes the redundant connection too). Debug log only; no special
  metric (Go increments none for a same-node duplicate).
- **Lock discipline:** `peers_lock` is a leaf lock held only across sync sections;
  documented inline so future edits don't introduce a nested `.await` or a
  `tracked_ips` ↔ `peers_lock` ordering hazard.
- **Self-connection / `node_id == my_node_id`:** out of scope here (handled, if at
  all, by the upgrader / handshake layer as before); this change only dedups
  distinct-peer duplicates.

## As-built notes

- **`Certificate` type path:** `admit_peer` uses `ava_crypto::staking::Certificate`,
  referenced by fully-qualified path in the signature; no additional `use` import
  was needed at the call sites.
- **T2 settle window:** no settle-window tweak was required — the 500 ms window
  held and tests passed consistently in 1.8–4.1 s (3–6× the window). However, T2
  did require a **production companion fix** that the plan did not anticipate:
  Task 1's atomic dedup surfaced a mutual-dial livelock. When two nodes dial each
  other simultaneously, TLS-1.3's client side completes before the server side, so
  each node admits its own outbound connection and rejects the peer's inbound —
  leaving the surviving connections half-dead. With a fixed retry delay both sides
  retry in lockstep forever (100% end-to-end failure, systematic). The fix was to
  port Go's retry jitter from `network/tracked_ip.go` `increaseDelay` — which the
  Rust port had omitted with a "jitter omitted" comment — into
  `crates/ava-network/src/network/tracked_ip.rs`, using Go's single-draw model
  (`delay *= (1+rand)`, near-cap `MAX*(3+rand)/4`, per-instance
  `jitter_seed = nanos^port`) so two peers' retry windows diverge and break
  lockstep. User explicitly chose this Go-style single-jitter approach over a
  deterministic NodeId tie-break alternative.
- M9.15 inbound at-most-once follow-up CLOSED; `BeaconManager` set demoted to
  defense-in-depth.
