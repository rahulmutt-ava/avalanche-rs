# M9.15 — Validate the v1/RSA TLS verifier fix live (staged)

**Date:** 2026-06-25
**Milestone:** M9.15 (interop hardening / live mixed network)
**Status:** Design approved; pending implementation plan

## Context

Commit `e06f0a0` landed the fix for the live `mixed_network` TLS stall: both
`ava-network` TLS verifiers now verify the TLS-1.3 `CertificateVerify` signature
via `verify_tls13_signature_with_raw_key` over the leaf SPKI, bypassing webpki's
X.509-v3 cert-version gate that rejected avalanchego's self-signed **v1 RSA**
staking certs with `UnsupportedCertVersion`. The leaf-key policy is unchanged
(security preserved). An in-process v1-RSA mutual-handshake test
(`crates/ava-network/tests/tls_v1_rsa_handshake.rs`, RSA-4096 fixture) proves
both directions offline.

What is **not** yet done: confirming the fix actually unblocks rustls↔Go interop
against a **real** avalanchego node, and tightening the gated live arms from
their RED-tolerant "the harness ran" assertions into genuine pass gates.

The repro matrix previously pinned the root cause empirically (see plan
M9.15 AS-BUILT notes): CELL2 (RSA, Go-side cert check disabled) was the decisive
cell — the Rust client rejected Go's RSA server cert with `UnsupportedCertVersion`;
CELL5 (ECDSA) succeeded. The fix targets exactly that rejection.

## Goal

Empirically confirm the verifier fix unblocks real rustls↔Go interop, in two
stages: first isolate the TLS layer against a real Go peer, then drive the full
mixed network and either green it or honestly diagnose + record the next stall.

## Stage 1 — TLS matrix goes green (isolated proof)

**What runs:** `tls_handshake_matrix_live` (`tests/differential/tests/tls_handshake_repro.rs`)
against a Go harness built from the current `~/avalanchego` checkout via
`tls_repro::build_go_harness` (nix go 1.26.3; `TLS_REPRO_GO` override available
if the default toolchain mismatches the checkout).

**Plumbing already correct:** `tls_repro::rust_client_upgrade` uses the
production `tls_config::client_config(id)`, which now carries the v1/RSA-accepting
verifier. No wiring change needed for the client path.

**Expected flip:** CELL2 (RSA, `verify=noop`) and CELL1 (RSA, `verify=staking`),
previously rustls-rejected with `UnsupportedCertVersion`, now complete the
handshake — the Rust client derives a NodeID and the Go side reports `ok=true`,
`version == 772` (TLS 1.3). CELL5 (ECDSA) already passed and must stay green.

**Test changes:** tighten the live arm from "assert only that both sides produced
a structured outcome" to **assert the RSA cells succeed**: the Rust client returns
a NodeID (`Ok`), and the parsed `GoOutcome` has `ok == true` and `version == Some(772)`.
This turns the diagnostic matrix into a real regression gate for the fix.

**Reverse-direction cell (included):** add one cell exercising Go *client* (real
RSA staking cert) → Rust *server* (`tls_repro::rust_server_upgrade`), so the Rust
**server** verifier is validated against a real Go RSA cert live — the
security-relevant inbound-peer path, not just the in-process fixture. The Go
harness already supports `--role=client` (commit `f0c127e`).

## Stage 2 — `mixed_network` end-to-end

**Hard prerequisite — rebuild `~/avalanchego`.** `Network::boot_mixed`
(`tests/differential/src/network.rs`) runs `scripts/check_oracle_binary.sh` as a
pre-gate and refuses to boot on a stale binary. The check currently FAILS: the
built binary is from `dbf0f71` but the checkout HEAD advanced to `2a93974`.
Rebuild with `cd ~/avalanchego && ./scripts/build.sh` (known-good in this
environment; the current binary was built today, just from a now-superseded
commit) and re-run the check until it prints `OK` (also asserts `rpcchainvm=45`).

**Env setup:** `AVALANCHEGO_PATH` must point at the **Go** binary
(`~/avalanchego/build/avalanchego`); it is currently mis-set to the Rust binary.
`avalanchers` is located via `$AVALANCHERS_PATH` or the conventional
`target/{release,debug}` path and must be built.

**What runs:** the `mixed_network` live arm — boot Go beacon (RSA `staker1` via
`local_staker(1)`) + Rust follower (fresh ECDSA cert via `generate_staker`),
wait up to 180s for the Rust node to bootstrap P/X/C, then drive a C-chain
transfer and assert both nodes settle to the same tip.

**Two acceptable outcomes (staged diagnose/record scope):**

- **(a) Green.** The Rust follower completes the TLS handshake with the Go
  beacon, exchanges PeerLists, bootstraps to Finished, and tracks the same tip.
  Tighten the arm's assertions and record that the live arm now passes (kept
  nightly-gated by default for cost).
- **(b) Next stall surfaces** — e.g. app-level Handshake message version, peer
  tracking, bootstrap connectivity gating, or the production boot-wiring gap
  (`RouterBridge` drops inbound in the real node boot path). Capture structured
  evidence (both sides' logs, the rung reached), write an honest root-cause note
  in `plan/M9-interop-hardening.md` mirroring the prior TLS diagnosis, file the
  next blocker as a follow-up task, and leave the arm gated. **Diagnose + record;
  do not fix-forward** — fixing the next layer is explicitly out of scope for
  this task.

## Deliverables

- Tightened `tls_handshake_repro.rs` live assertions, plus the reverse-direction
  cell wired through `tls_repro.rs`.
- A `plan/M9-interop-hardening.md` AS-BUILT note recording the live result of
  both stages.
- If Stage 2 hits a new blocker: a pinned root-cause note + a clear next-task
  handoff.
- No production-code changes expected unless Stage 1/2 reveals a verifier
  regression.

## Testing

Stage 1 and Stage 2 are `#[cfg(feature = "live")] #[ignore]` arms, run via
`cargo nextest run -p ava-differential --features live -- --ignored`, gated on
`AVALANCHEGO_PATH`. The offline arms (`offline_gate_and_parse`,
`mixed_network_replay_is_deterministic`) stay green and unchanged. The in-process
`crates/ava-network/tests/tls_v1_rsa_handshake.rs` remains the CI regression
floor for the verifier fix.

## Out of scope

- RSA staking-key *loading* (`Identity::from_pem` PKCS#1 support) — separate
  deferred gap; the Rust follower uses an ECDSA cert.
- Fixing whatever next-layer stall Stage 2 may reveal (would be its own task).
- Un-gating the live arms in per-PR CI (they remain nightly/operator-gated).
