# M9.15 rung 4: local-network genesis identity mismatch (Rust ≠ Go)

**Date:** 2026-07-15 · **Branch:** `m9.15-live-mixed-net-v07` (Task-8 loop,
iteration 4) · **Status:** approved for plan

## 1. Problem

Live `mixed_network` run 7 (all prior rungs fixed) reaches the C-transfer
check and fails: Go mines C height 1; the Rust follower stays at 0 for 60s.
Investigation (`.superpowers/sdd/task7-rung4-investigation.md`) proves the
follower's NormalOp discovery loop WORKS — it pull-queries, receives chits
naming the new block `QfBiv4` (acceptedHeight=1), and fetches its 791-byte
proposervm container — but adoption dead-ends: the block's parent is the
**network's** coreth genesis (`jXM4…`, ethhash `608ddb…07b8b1`, = go1's
logged "read last accepted hash"), while the follower materialized a
**different** local genesis (`2eQSeZ…`, ethhash `d8a07966…`). The fetched Go
genesis is height-0 = "decided" locally, so it is silently skipped
(`is_decided` → `should_issue_block=false`, engine.rs:647-666) and `QfBiv4`
is forever un-issuable. The P-chain logs show the identical mismatch
signature (Go P gossip names blocks the follower reports "not found"; P
genesis IDs differ) — bootstrap only "succeeded" because equal heights (0=0)
short-circuit before any genesis-ID comparison. X presumably likewise.

Related earlier finding now explained: Go rejecting the follower's served
container with `invalid difficulty: 0` (Task 6) — the follower serves its own
divergent genesis header where the network expects coreth's.

## 2. Goal

For `network-id=local` (and by construction fuji/mainnet), the Rust node's
materialized P/X/C genesis blocks are **byte- and ID-identical** to
avalanchego's (oracle `96897293a2`): same container bytes, same block IDs,
same C ethhash root. The mixed-net follower can then root network blocks.

## 3. Approach

Byte-diff first, then fix at the source:
1. Obtain both C genesis containers: the follower already fetched Go's 598-byte
   genesis in run 7 (and go1 logs its hash); the Rust one comes from
   `EvmVm::from_genesis`. Diff the decoded headers field-by-field (state root,
   difficulty, gasLimit, timestamp, extraData, base fee…) to name the exact
   divergent field(s). Same exercise for P (ava-genesis P genesis bytes vs
   Go's `2Z95Kpsd…` / follower's differing ID) — the root cause may be shared
   (genesis JSON source) or per-chain (C header construction).
2. Fix where the divergence lives: the local-network genesis inputs in
   `ava-genesis` (must mirror avalanchego `genesis/genesis_local.go` +
   embedded C-chain genesis JSON) and/or `EvmVm::from_genesis` header
   construction (M6.8). The M6 recorded-oracle `cchain_genesis_root` test
   passed — so either its input JSON differs from the live local genesis, or
   the divergence is in fields outside that test's scope; the plan must
   reconcile why it was green.
3. Pin with Go-oracle goldens: a test asserting the local-network P/X/C
   genesis IDs (and C ethhash root) equal values extracted from the oracle
   (bytes already in hand from run 7; re-emit via ~/avalanchego if needed for
   provenance).

## 4. Acceptance

1. RED goldens (current divergent IDs) → GREEN after fix; provenance recorded.
2. Solo live probe: follower's C genesis hash == `608ddb…07b8b1` shape check
   via eth_getBlockByNumber(0) against a spawned release binary, compared to
   the Go oracle's value for network-id=local.
3. Live `mixed_network` re-run: follower follows the C transfer to height 1
   (or fails demonstrably later); `mixed_network_single_beacon` still passes.
4. Workspace tests + lint green; any golden regenerations carry provenance
   notes.

## 5. Out of scope

Engine logging of parse/verify failures (investigation found errors swallowed
— record as follow-up); per-chain log wiring; the P-chain gossip "not found"
noise beyond what the shared genesis fix resolves.
