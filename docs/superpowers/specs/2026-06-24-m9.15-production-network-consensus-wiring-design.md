# M9.15 — Production network→consensus wiring

**Date:** 2026-06-24
**Status:** Design (approved for plan)
**Area:** `crates/avalanchers` (chain wiring), `crates/ava-node` (node assembly, networking), `crates/ava-engine` (router)
**Predecessor:** `2026-06-23-m9-networked-bootstrap-design.md` (Tasks 1–7, merged `39c533a`)

## 1. Problem

The two-Rust-node bootstrap proven by `crates/avalanchers/tests/networked_bootstrap.rs`
exercises the network→consensus seam through a **hand-built** harness, not the
production node-assembly boot path. In a real `avalanchers` process the loop is
open at both ends:

- `init_networking` builds a `RouterBridge` and `init_chain_manager` fills its
  engine-router slot with the node's **one** shared `ChainRouter`
  (`chain_manager.rs:385` — `router_bridge.set_engine_router(chain_router.clone())`).
- But `drive_startup_chains_with_db` → `run_queued_chains_with_db` boots every
  chain over the **in-process `RecordingSender` loopback** and over a **per-chain
  `ChainRouter` that `boot_chain_over_network`/`boot_chain` build internally**.

Consequently, on a live node:

- inbound p2p is decoded by `RouterBridge` and handed to the shared
  `ChainRouter` — which has **no chains registered into it** (each chain
  registered into its own loopback-path router), so inbound ops are dropped; and
- chain engines send outbound ops to their loopback `RecordingSender`, never to
  real peers over `networking.net`.

This is the documented follow-up recorded in commit `39c533a` and in
`plan/M9-interop-hardening.md` (the M9.15 deferral note around line 1330).

## 2. Goal

Boot production chains so that:

1. each booted chain registers its handler into the node's **shared**
   `ChainRouter` (the one `RouterBridge` already routes inbound to); and
2. each chain sends outbound via a real `OutboundSender` over `networking.net`.

Delivered in two sequenced phases:

- **Phase 1 (solo/beaconless):** keep behavior identical — P/X/C still
  short-circuit `Bootstrapping → NormalOp` — but over the production seams
  (shared router + real `OutboundSender`), with a CI test proving the inbound
  loop closes.
- **Phase 2 (beaconed follower):** a node with configured bootstrap beacons
  actually bootstraps from them over the network, proven by a **full two-`Node`
  convergence** integration test on localhost TLS driven through the production
  `drive_startup_chains` path.

SAE dispatch remains the separate documented deferral (M7.21/M7.26 seams +
genesis-bytes materialization) and is out of scope here.

## 3. Central decision: one shared router (Approach A)

`ChainRouter` is a `chain_id → handler` multiplexer; there must be exactly one
per node. The refactor threads the node's single `Arc<ChainRouter>` (already
produced as `ChainManagerInit.chain_router`) into the network boot. Each chain
registers into it via the existing `create_snowman_chain` `add_chain` call; the
`RouterBridge` already points at it, so the inbound loop closes with no extra
wiring. This matches Go's single-process-router model.

Rejected alternatives:

- **B — per-chain routers + multiplexing `RouterBridge`** (`chain_id→router` map
  on the bridge): duplicates the multiplexing `ChainRouter` already performs,
  diverges from Go, adds state. Rejected.
- **C — register-after-boot merge** (boot builds its own router, then merge its
  one registration into the shared router): request/timeout state splits across
  two routers. Rejected.

## 4. Phase 0 — enabling refactor

1. **Expose the shared router.** Retain `chain_manager_init.chain_router` as a
   `Node` field (`pub chain_router: Arc<ChainRouter>`) so the binary's
   chain-creation driver can hand it to the boot path. (Today it is created in
   `init_chain_manager`, installed into `RouterBridge`, and otherwise dropped.)

2. **Parameterize `boot_chain_over_network`.** Change it to accept, instead of
   building internally:
   - `router: Arc<ChainRouter>` — the node's shared router (brings the
     production `AdaptiveTimeoutManager` built with config timeouts + `RealClock`);
   - `clock: Arc<dyn Clock>` — the VM clock (production passes `RealClock`); and
   - it continues to take the `Allower` for the per-chain `OutboundSender`.

   The internal `MockClock::at(UNIX_EPOCH)` + fresh `ChainRouter` construction is
   removed. The `OutboundSender` is still built per-chain from
   `(network, allower, router, chain_id, subnet_id, initial_timeout)` — each
   chain gets its own sender over the *shared* router + network.

3. **Mechanical test updates.** The three call sites that rely on the old
   self-contained signature — `networked_bootstrap.rs` and the
   `outbound_sender_boot` / `beacon_connectivity_gate` tests — pass an
   explicitly-constructed `ChainRouter` + `MockClock` and (for
   `networked_bootstrap`) install that router on their per-node `RouterBridge`
   as before. Behavior unchanged for those tests.

## 5. Phase 1 — solo/beaconless over production seams

1. **New driver `run_queued_chains_over_network`** (sibling of
   `run_queued_chains_with_db`):
   `(manager, network_id, base_db, net: Arc<dyn Network>, allower, router: Arc<ChainRouter>, gate)`.
   For each queued P/X/C chain it calls `boot_chain_over_network` with the shared
   router + a real `OutboundSender` over `net`, with `beacons = Some(empty)`
   ⇒ `Bootstrapping → NormalOp` short-circuit (identical end-state). It keeps the
   `set_bootstrapped_reporter` install so `info.isBootstrapped(chain)` stays a
   live read of `ctx.state == NormalOp`.

2. **`drive_startup_chains_with_db` beaconless branch** calls
   `run_queued_chains_over_network`, threading:
   - `node.networking.net` (the real `NetworkImpl` as `Arc<dyn Network>`);
   - a primary-network `AllowAll` allower (P/X/C are primary-network; subnet
     `Allower` deferred);
   - `node.chain_router` (the shared router); and
   - `node.networking.on_sufficiently_connected` as the gate (pre-fired `true`
     for a solo node, so start is immediate).

3. **CI test (loop-closing proof)** in `crates/avalanchers/tests/`: boot the solo
   P-chain through the production path against a recording/mock `Network`;
   synthesize an inbound `GetAcceptedFrontier` from a fake peer and deliver it
   through `RouterBridge`'s `ExternalHandler`; assert the chain's `Getter`
   answers `AcceptedFrontier` out through the `OutboundSender` (observed on the
   recording `Network`). This proves inbound routing **and** outbound send across
   the production seam in one test, without a Go node.

## 6. Phase 2 — beaconed follower in production

1. **`drive_startup_chains_with_db` beaconed branch** (today returns an empty
   vec): boot each critical chain with
   `beacons = Some({ b.id → 1 for b in config.bootstrap_config.bootstrappers })`,
   gated on `on_sufficiently_connected`, over the same
   `run_queued_chains_over_network` driver.

2. **Dial the configured beacons.** The binary/init must call
   `net.manually_track(b.id, b.ip)` for each configured bootstrapper. This is a
   real gap: `init_networking`'s `BeaconManager` only *counts* beacon
   connections to drive `on_sufficiently_connected`; it does not dial. The
   backoff dialer (Task 1, merged) connects to tracked peers. Wire the tracking
   from the parsed `BootstrapConfig.bootstrappers` (`{ id: NodeId, ip: SocketAddr }`).

3. **CI test — full two-`Node` convergence** (`crates/avalanchers/tests/`):
   stand up **two real `ava_node::Node` assemblies** on localhost TLS:
   - **Beacon node** — no configured bootstrappers (beaconless) ⇒ its critical
     chains short-circuit to `NormalOp` and its `Getter` answers `Get*`.
   - **Follower node** — the beacon as its sole configured bootstrapper.

   Drive both nodes through the production `drive_startup_chains` path (not the
   hand-built harness), drive both network dispatch loops, and assert the
   follower reaches `NormalOp` (bootstrapper `Finished`) and converges on the
   beacon's tip (same last-accepted block id + height) for the platform chain.
   This is the live two-binary arm minus the Go peer; the live Go arm stays
   nightly-gated.

   Note: full `Node::new` binds real staking/API listeners and spins API
   servers; this test is heavier and may run serially / with generous timeouts.
   The Phase 1 focused loop-closing test remains the fast-feedback gate.

## 7. Risks & deferrals

- **Clock change (in scope):** production network boot switches the per-chain
  VM clock and timeout manager from `MockClock`-at-epoch to `RealClock` + the
  shared `AdaptiveTimeoutManager`. Inert for the solo short-circuit; correct for
  the beaconed path's request timeouts. Tests pass `MockClock` explicitly.
- **Subnet `Allower` deferred:** P/X/C are primary-network, so `AllowAll` is
  correct; a subnet-membership allower is a later subnet-support item.
- **SAE dispatch:** unchanged deferral (M7.21/M7.26 seams + genesis-bytes
  materialization); the `run_queued_chains` SAE branch stays a `warn`.
- **Two-`Node` test weight:** the Phase 2 integration test assembles two full
  nodes; if it proves flaky under CI port/timing constraints, gate it behind a
  serial/`#[ignore]`-able marker while keeping the Phase 1 fast test always-on.

## 8. Definition of done

- `drive_startup_chains_with_db` boots P/X/C over the shared `ChainRouter` +
  real `OutboundSender`; `node.chain_router` is the router `RouterBridge` routes
  inbound to (asserted reachable).
- Phase 1 loop-closing test green: inbound `GetAcceptedFrontier` through
  `RouterBridge` reaches a production-booted chain's `Getter` and the reply
  leaves via the `OutboundSender`.
- Phase 2 two-`Node` convergence test green: follower bootstraps from the beacon
  through the production path to `NormalOp` and converges on the beacon's tip.
- `boot_chain_over_network` and the three existing call sites compile against the
  new signature; `networked_bootstrap.rs` still passes.
- `lint-all` + `test-unit` green; no behavior change for a solo `--network-id=local`
  node (P=X=C `isBootstrapped` still flips true).
