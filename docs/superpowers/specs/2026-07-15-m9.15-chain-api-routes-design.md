# M9.15 rung 2: live node serves no chain APIs (`/ext/bc/*` → 404)

**Date:** 2026-07-15 · **Branch:** `m9.15-live-mixed-net-v07` (Task-8 loop,
iteration 2) · **Status:** approved for plan

## 1. Problem

With the frontier stall resolved (stale-binary rung, `ece6932`), live
`mixed_network` advances to `boot_mixed` PASS — the Rust follower bootstraps
P/X/C against the 5-Go quorum — then fails `await_all_connected`:
`Observation::collect` returns <5 fields for the Rust node because every
chain-scoped API call fails. Verified on a solo HEAD follower
(`.superpowers/sdd/` probe, 2026-07-15):

- `/ext/info` → 200 (`info.getNodeID`, `info.getNodeVersion` fine)
- `/ext/bc/P` → **404**; `/ext/bc/C/rpc` → **404** (so `platform.getHeight`,
  `platform.getCurrentValidators`, `eth_blockNumber` all fail)

Cause: the live chain-boot path (`run_queued_chains*` /
`boot_chain_over_network*` in `crates/avalanchers/src/wiring/chains.rs`) boots
VMs to NormalOp but never registers their HTTP handlers with the node's
`ava_api::Server`. All the mount machinery already exists and is tested at the
ava-api level (M8 `register.rs`: `/ext/bc/<chainID>/<ext>` mounts, bc/P-style
aliases, not-bootstrapped 503 layer). Go parity: chain creation calls
`VM.CreateHandlers` and mounts each handler via the API server
(`chains/manager.go`). This is the M8.31-deferred wiring surfacing live.

The same `Observation::collect` is the milestone's final fork-check, so this
rung gates everything after bootstrap.

## 2. Goal

A live `avalanchers` node serves its booted chains' APIs: `platform.*` on
`/ext/bc/P`, the EVM RPC on `/ext/bc/C/rpc`, X on `/ext/bc/X` — same paths,
aliases, and 503-while-bootstrapping semantics Go serves, using the existing
ava-api registration seam.

## 3. Approach (single reasonable one — wiring, not design)

In the live boot path, after each chain's VM is constructed (P, X, C — and any
future vm_id branch), obtain its HTTP handlers (`create_handlers()` on the VM,
M8 seam) and register them through the existing ava-api chain-registration
entrypoint with the chain's ID + primary alias, gated by the chain's
bootstrapped signal for the 503 layer. Requires threading the node's
`api_server` (and alias registry) into `run_queued_chains_with_db` /
`drive_startup_chains_with_db` — an additive parameter on an internal seam,
mirroring how `base_db` was threaded in wave-2026-06-19d.

Explicitly NOT in scope: response-shape parity fixes inside individual RPC
methods (if the live run then trips on a shape mismatch, that is the next
rung, observed with working routes); per-chain log wiring; the Task-6
`invalid difficulty` finding.
