# M9.15 — Live `mixed_network` validation (firewood v0.7.0)

**Date:** 2026-07-05
**Milestone:** M9 (interop hardening), task M9.15 live two-binary arm
**Status:** design approved, pre-implementation

## Objective

Execute the nightly-gated live two-binary `differential::mixed_network` arm
against a real Go `avalanchego` oracle and prove that the merged
bootstrap-failure-accounting + beacon-gate-race fixes make the Rust follower
actually bootstrap P/X/C from Go. The last live run (rung-3, memory
`m9.15-live-mixed-net-blocker`) reached Stage-1 green (5-Go cluster bootstraps)
but stalled at Stage-2: the Rust follower completed TLS + app-Handshake yet
emitted no `GetAcceptedFrontier`/`Ancestors` and never left connect→
bootstrap-start. The production cause was diagnosed and fixed offline; this run
is its live validation.

The oracle is rebuilt at `~/avalanchego` HEAD `d8a8473b` (up from the reviewed
pin `cbea6289`). That drift bumps firewood ffi `v0.6.0 → v0.7.0`, so to keep the
C-chain state-root comparison valid the Rust workspace firewood pin is bumped to
match. This makes the effort a code change plus a validation-and-debug loop.

## Scope

This is **validation-first**, not fresh feature work. Three parts, in order.

### Part A — Firewood v0.6.0 → v0.7.0 bump (prerequisite code change)

- Update the four tag declarations to `tag = "v0.7.0"`:
  - `crates/ava-evm/Cargo.toml` (`firewood`, `firewood-storage`)
  - `crates/ava-merkledb/Cargo.toml` (`firewood`, `firewood-storage`)
- `cargo update -p firewood -p firewood-storage`; build and resolve any v0.6→v0.7
  API breakage in `crates/ava-merkledb/src/firewood/*` and the `ava-evm` firewood
  wiring. The API delta is unknown until the build runs.
- Re-verify firewood-dependent goldens:
  - `crates/ava-merkledb/tests/golden_firewood_ethhash.rs` (ethhash root vector)
  - the M6 C-chain `state_root` recorded-oracle tests in `ava-evm`.
- **Root-move policy:** the ethhash root should move only if v0.7 intentionally
  changed hashing. If it moves, regenerate the golden vector from v0.7.0 and
  record provenance in `crates/ava-merkledb/tests/vectors/firewood/ethhash/_provenance.md`.
  An *unexpected* move is a red flag to investigate before accepting, not an
  auto-regenerate.
- `deps-tidy` (commit `Cargo.lock` + `MODULE.bazel.lock`) + `bazel-check-metadata`.

### Part B — Oracle rebuild + drift fold

- Rebuild `~/avalanchego` at `d8a8473b` via the toolchain-correct path: **mise go
  1.25.10 on PATH + GOROOT** (nix go 1.26.3 silently fails to relink — known
  gotcha, memory `go-avalanchego-oracle-binary`).
- `./scripts/check_oracle_binary.sh` must print `OK` and assert `rpcchainvm=45`.
- Light `avalanchego-sync` pass. The drift is 22 commits (2026-06-25 →
  2026-07-02) but rpcchainvm protocol (45), version string, handshake, and bloom
  constants are **confirmed unchanged** in-range; only firewood ffi moved
  (v0.6.0 → v0.7.0). Fold the firewood delta as an Upstream-delta callout and
  bump the specs reviewed-through pin `cbea6289 → d8a8473b` in `specs/README.md`.
  A full re-review is not warranted given constants are unchanged.

### Part C — Live run + debug loop

- Run `mixed_network_single_beacon` first (required_conns=1 bisection probe):
  does the follower bootstrap against a single real Go beacon? Isolates
  engine/gate/frontier from multi-beacon connectivity.
- Then the full `mixed_network` (5-Go quorum + Rust follower; drive one C-chain
  transfer on the Go validator; assert both nodes reach the same C height and
  byte-identical normalized `Observation`).
- Invocation: `AVALANCHEGO_PATH=~/avalanchego/build/avalanchego cargo nextest run
  -p ava-differential --features live -- --ignored`.
- **If a new production blocker surfaces** (the established "rung" pattern — every
  prior live run has revealed one), branch into a design → plan → TDD-fix cycle
  for that specific blocker, then re-run. If green, un-gate what is safely
  CI-runnable and record AS-BUILT in `plan/M9-interop-hardening.md` + memory.

## Verification

- **Checkpoint:** `mixed_network_single_beacon` green ⇒ the follower bootstraps
  against real Go (the primary open question) — a meaningful result even if the
  full arm then reveals a fresh rung.
- **Success:** full `mixed_network` green — no fork, same tip, matching
  normalized observation across the mixed net.
- Workspace stays buildable-and-green throughout: after Part A, full
  `test-unit` + `lint-all` must pass before touching the oracle.

## Risks

1. **Firewood v0.7 API breakage** beyond a version bump. Mitigation: isolate the
   bump, build incrementally, resolve breakage in the two firewood-consuming
   crates only.
2. **v0.7 moves the ethhash root**, invalidating M6 recorded-oracle C-root
   vectors. Mitigation: regenerate against v0.7.0 and re-verify with provenance;
   do not blind-accept an unexpected move.
3. **Live run reveals a fresh production blocker** (likely, by track record).
   Handled by the Part-C debug-loop branch — not a failure of this plan. Each
   rung is its own design→plan→fix sub-cycle.

## Non-goals

- No full re-review of the 22-commit upstream drift (constants confirmed
  unchanged; light fold only).
- No attempt to un-gate the live arm into per-PR CI — it stays nightly/operator-
  gated; only newly-provable CI-runnable properties get un-gated.
- No other M9 tasks (M9.17/19/20/21/23) — those are separate threads.
