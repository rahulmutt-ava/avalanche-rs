# M9.15-next — Rust outbound `Handshake.knownPeers` bloom filter

**Date:** 2026-06-26
**Milestone:** M9.15-next (next live `mixed_network` blocker; filed in `plan/M9-interop-hardening.md`)
**Crates:** `ava-network` (primary), `ava-message` (already wired)
**Specs:** `05` §3.5 (peer-list gossip / IP tracking), `26` §9 (handshake compatibility)
**Status:** approved design

## Problem

The Rust follower's outbound application-layer `Handshake` message carries a
malformed `knownPeers` bloom filter. `build_handshake`
(`crates/ava-network/src/peer/peer.rs:450-451`) passes empty byte slices `&[]`
for both `known_peers_filter` and `known_peers_salt`. avalanchego's
`bloom.Parse` reads byte 0 as `num_hashes`; on an empty slice that fails with
`errInvalidNumHashes` ("invalid num hashes"). The Go beacon therefore rejects
every Rust `Handshake` (`peer/peer.go:940 malformed message
{field:"knownPeers.filter", error:"invalid num hashes"}`), never registers the
Rust node as a peer (`Peers:0` throughout), so bootstrap can never seed a
frontier. This is the live blocker that remains after the TLS-1.3 verifier fix
(`e06f0a0`) was confirmed live (`2ad33d2`).

The Rust side today has only a **read** filter (`ReadFilter` in
`network/bloom.rs`, the M2.17 port), and `IpTracker` holds no bloom state at
all — so there is no encode/marshal path to produce a valid filter.

## Goal

Emit a structurally valid, Go-faithful `knownPeers` bloom filter + 32-byte salt
on every outbound `Handshake` (and `GetPeerList`), mirroring avalanchego's
`network/ip_tracker.go` bloom lifecycle, so the Go beacon registers the Rust
peer (`Peers:1`) and bootstrap proceeds. Make the produced filter and the
already-consumed filter mutually consistent (fix a latent read-side key
divergence).

## Non-goals (deferred follow-ups)

- Periodic `ResetBloom` timer and auto-reset when `count ≥ max_bloom_count`
  (M2.20 timer wiring). The first bootstrap handshake carries an empty filter
  regardless of these, so they do not gate the live arm.
- `ip_bloom` Prometheus gauges (`specs/18`).

These are documented inline where the seams land.

## Background — the Go encoding (anchored in `~/avalanchego`)

Wire format (`utils/bloom/filter.go` `marshal`), identical to what the existing
Rust `ReadFilter::parse` already reads:

```text
num_hashes (1 byte) || hash_seeds (num_hashes * 8, BE) || entries (>= 1 byte)
```

Constants (`utils/bloom/filter.go`): `minHashes=1`, `maxHashes=16`,
`minEntries=1`, `bitsPerByte=8`, `hashRotation=17`. Seeds come from
`crypto/rand`.

`ip_tracker.go` bloom lifecycle:
- `saltSize = 32`, `minCountEstimate = 128`, `targetFalsePositiveProbability = .001`,
  `maxIPEntriesPerNode = 2`.
- `resetBloom()`: `count = max(maxIPEntriesPerNode*len(tracked), minCountEstimate)`
  → `numHashes, numEntries = bloom.OptimalParameters(count, .001)` → `bloom.New(...)`
  + fresh 32-byte salt → re-`Add` every tracked node's `ip.GossipID` →
  `maxBloomCount = bloom.EstimateCount(numHashes, numEntries, maxFalsePositiveProbability)`.
- `Bloom() ([]byte, []byte)` returns `(bloom.Marshal(), bloomSalt)`; the peer
  sends this as `KnownPeers()`.
- The bloom key is `ip.GossipID` = `ComputeHash256Array(packer{NodeID || timestamp})`
  (`utils/ips/claimed_ip_port.go`), **not** the raw NodeID.

`OptimalParameters` / `OptimalEntries` / `OptimalHashes` / `EstimateCount`
(`utils/bloom/optimal.go`) use `math.Log`/`math.Ceil`/`math.Pow` over
`ln2`/`ln2Squared`.

## Design

### Component 1 — write-side bloom in `network/bloom.rs`

Extend the existing read-only module with the encode halves (port of
`utils/bloom/filter.go` + `optimal.go`):

- `pub struct Filter { num_bits: u64, hash_seeds: Vec<u64>, entries: Vec<u8>, count: usize }`.
  Lock-free: it lives inside `IpTracker`'s existing `Mutex<Inner>`, so it does
  not need Go's per-`Filter` `RWLock`.
- `Filter::new(num_hashes, num_entries) -> Result<Filter, BloomError>` — validates
  `num_entries >= MIN_ENTRIES`; fills `num_hashes * 8` seed bytes from
  `ring::rand::SystemRandom` (the crate's existing CSPRNG, used by
  `identity.rs`/`peer/ip.rs`; matches Go `crypto/rand`), decoding each seed BE.
- `Filter::add(&mut self, mut hash: u64) -> bool` — port of Go `Add`
  (rotate-left-17 / XOR seed / set bit, increments `count` when newly added).
- `Filter::add_key(&mut self, key: &[u8], salt: &[u8]) -> bool` = `add(hash(key, salt))`
  (Go free function `bloom.Add`); reuses the existing `hash()`.
- `Filter::marshal(&self) -> Vec<u8>` — `[num_hashes] || seeds(BE) || entries`;
  round-trips through `ReadFilter::parse`.
- `Filter::count(&self) -> usize`.
- `optimal_parameters(count, fpp) -> (usize, usize)`, `optimal_entries`,
  `optimal_hashes`, `estimate_count` — verbatim float ports of `optimal.go`.

> **Float note.** `optimal.go` uses floating-point. This is *gossip filter
> sizing*, not a codec or consensus path: outputs are bounded integers and the
> two peers parse each other's filters independently (they never need to agree
> on a size). It is therefore exempt from the no-float-in-consensus rule
> (`00` §). Confirm `ava-network` is not float-denied by `clippy.toml` / lib
> inner attrs; if it is, isolate these four functions with a scoped `#[allow]`
> + this rationale.

### Component 2 — `IpTracker` owns the bloom (`network/ip_tracker.rs`)

- `Inner` gains: `bloom: Filter`, `bloom_salt: Vec<u8>` (32 bytes),
  `bloom_additions: BTreeMap<NodeId, usize>`, `max_bloom_count: usize`.
- `IpTracker::new()` becomes an explicit constructor that calls `reset_bloom()`
  to seed the initial valid empty filter + salt. (`#[derive(Default)]` is
  removed; any `IpTracker::default()` callers move to `new()`.)
- `reset_bloom(&self)` (private, takes the inner lock or operates on `&mut Inner`):
  `count = max(2 * claims.len(), 128)` → `optimal_parameters(count, 0.001)` →
  `Filter::new(...)` + a fresh 32-byte salt from `SystemRandom` → re-add every
  tracked claim's GossipID → `max_bloom_count = estimate_count(...)` → clear
  `bloom_additions`.
- On `add_claimed_ip_port` (after inserting/replacing the claim): if
  `bloom_additions[node] < maxIPEntriesPerNode (2)`, `bloom.add_key(gossip_id, salt)`
  and bump the per-node count. (Auto-reset when `count >= max_bloom_count` is a
  documented deferred follow-up — for the bootstrap arm the filter never
  saturates.)
- `manually_track` (bootstrappers, no verified IP / `timestamp == 0`,
  `cert_der` empty) is **not** added to the bloom — mirrors Go's
  `trackedNode.ip == nil` skip.
- `pub fn bloom(&self) -> (Vec<u8>, Vec<u8>)` returns
  `(self.bloom.marshal(), self.bloom_salt.clone())` (Go `Bloom()` / the peer's
  `KnownPeers()`).

### Component 3 — GossipID fix (corrects a latent read-side divergence)

Add `fn gossip_id(node: &NodeId, timestamp: u64) -> [u8; 32]` =
`sha256(node.as_bytes() || timestamp.to_be_bytes())` (port of Go
`ClaimedIPPort.GossipID`; the Go packer packs the fixed NodeID bytes then the
BE u64 timestamp). Use it in **both**:

1. the bloom seeding in Component 2, and
2. the existing `IpTracker::peers()` membership check, which today wrongly uses
   `filter.contains_key(node.as_bytes(), salt)` — change to the claim's
   `gossip_id(node, claim.timestamp)`.

This makes the filter we produce and the filter we consume mutually consistent
and both match Go. It is a behavior change to `peers()`; existing `ip_tracker`
tests that exercise dedup are updated to seed the filter with GossipID.

### Component 4 — wire into the outbound messages (`peer/peer.rs`)

`build_handshake`: replace the two `&[]` arguments with
```rust
let (filter, salt) = self.cfg.ip_tracker.bloom();
```
and pass `&filter, &salt`. `PeerConfig.ip_tracker: Arc<IpTracker>` already
exists (`config.rs:110`), so no new plumbing. Apply the same change to the
periodic `GetPeerList` send path (the M2.17 `builder` already takes filter+salt
parameters).

## Data flow

```
build_handshake / GetPeerList send
   └─ self.cfg.ip_tracker.bloom() ─→ (marshal() bytes, salt)
         marshal() = [num_hashes] || seeds(BE) || entries   (valid Go bloom.Parse input)
   └─ creator.handshake(... &filter, &salt ...) ─→ p2p::BloomFilter { filter, salt }

inbound PeerList → add_claimed_ip_port → insert claim
   └─ bloom.add_key(gossip_id(node, ts), salt)   (per-node cap 2)

inbound GetPeerList → peers(their_filter, their_salt)
   └─ skip claims whose gossip_id(node, claim.ts) is contains_key in their_filter
```

## Error handling

- `Filter::new` returns `BloomError::TooFewEntries` / `TooFewHashes` /
  `TooManyHashes` for out-of-range params (reuses the existing enum). For the
  fixed `count=128, fpp=.001` inputs these never trigger, but the constructor
  stays total.
- `reset_bloom` only fails if `Filter::new` fails (impossible for the computed
  params) or the CSPRNG fails; surface a `Result` and map a CSPRNG failure to
  the existing error path (do not panic — library code, `unwrap`/`expect`
  denied).
- `bloom()` is infallible (reads already-constructed state).

## Testing (TDD)

- **Unit (`bloom.rs`):**
  - `Filter::marshal` round-trips through `ReadFilter::parse` (no error; seeds +
    entries preserved).
  - `add` then `contains` is true for the added key; an unrelated key is
    (almost surely) false on a large empty filter.
  - `optimal_parameters(128, 0.001)` equals Go's fixed output (hardcode the
    expected `(num_hashes, num_entries)` computed from `~/avalanchego`).
  - `bloom()` output is Go-`bloom.Parse`-structurally-valid: `num_hashes ∈ [1,16]`,
    `len == 1 + num_hashes*8 + entries_len`, `entries_len >= 1`.
- **Integration (the red→green driver, `ava-network`):** a `build_handshake`
  test asserting the emitted `Handshake.known_peers.filter` parses via
  `ReadFilter::parse` without error and `salt.len() == 32` — the exact decode Go
  performs. Goes **red** against today's `&[]` (parse → `InvalidNumHashes`).
- **`peers()` dedup:** a test seeding a filter with a node's GossipID and
  asserting that node is skipped, while a node not in the filter is returned —
  proves Component 3.
- **Wire-identity (extend M9.13 matrix):** structural diff of the Rust-emitted
  filter vs a Go-emitted one. Random seeds/salt preclude byte-equality, so
  assert parse-validity + range + a seeded-key membership probe (Rust filter
  `contains` a key Go's filter would, given the same seeds/salt — or assert both
  parse and the empty-filter case matches Go `bloom.EmptyFilter` shape).
- **Live (`mixed_network`, `#[cfg(feature="live")] #[ignore]`, nightly,
  recorded):** re-run → expect Go logs `Peers:1`, no "invalid num hashes",
  bootstrap proceeds. Honor the pre-gate `check_oracle_binary.sh` (pin
  `cbea62895c`, `rpcchainvm=45`) before any live run.

## Risks

- **Float lint on `ava-network`.** Mitigation in the Float note above; isolate
  with a scoped `#[allow]` + rationale if needed.
- **`peers()` behavior change** may break existing `ip_tracker` dedup tests —
  update them to seed with GossipID (expected, part of Component 3).
- **`IpTracker::default()` removal** — sweep callers (`testutil.rs`, network
  bring-up) to `IpTracker::new()`.
- The live arm stays gated; this design unblocks it but the run is on the
  nightly budget and depends on the Go oracle binary matching its pin.
