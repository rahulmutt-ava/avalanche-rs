# M9.15 — rustls↔Go TLS 1.3 handshake repro & diagnosis

**Date:** 2026-06-24
**Status:** Design (approved for plan)
**Area:** `tests/differential/go-oracle/` (new Go harness), `crates/ava-network` (gated live test), `crates/ava-crypto/src/staking` (only if a cert-template fix is localized)
**Predecessor:** `2026-06-24-m9.15-production-network-consensus-wiring-design.md` (production wiring merged); the live `mixed_network` arm AS-BUILT root-cause note in `plan/M9-interop-hardening.md` (2026-06-23, ~line 1509)

## 1. Problem

The live two-binary `differential::mixed_network` arm — the last real blocker
for live Go interop and the `16` §5 drop-in definition-of-done — fails at the
TLS layer, *below* the Avalanche app-level handshake. From the 2026-06-23
investigation:

- The Rust follower dials the Go beacon as the **TLS client**, reaches
  rustls' `Attempting client auth` (after Go's `CertificateRequest`), and then
  **stops**; the next log line is a fresh redial. The TLS handshake never
  completes (`upgrader.rs:142` `TLS upgrade complete` never fires).
- The Go beacon shows **zero** `peer`/`upgradeConn`/inbound-conn events — it
  drops the connection *before* handing off to its peer upgrader, so Go's
  server-side TLS handshake error is never surfaced anywhere observable.
- Neither side logs a TLS alert. It presents as a **stall**, not a clean reject.

The signed-IP and `Handshake`-wire hypotheses were ruled out (no app-level code
path is reached). The failure is strictly a **rustls-client ↔ Go
`crypto/tls`-server TLS 1.3 mutual-auth interop** problem, and the single
missing signal is **Go's server-side handshake error** — invisible in the
full-node path because Go drops the conn before its peer upgrader runs.

The full-node path is also hampered by two separate defects noted on 2026-06-23:
native `ava_*` `tracing` events do not reach the `LogFactory` sinks (only the
`log`→`tracing`-bridged `rustls` records land), and (since fixed) the
`handle_dial` upgrade `Err` arm now logs (`net_impl.rs:254`). Diagnosing in the
full node therefore inherits a logging-wiring fix as a prerequisite.

## 2. Goal

Get **ground truth** on the exact failing TLS rung and the root cause, fastest,
by reproducing the handshake **outside** the full node — pairing our real
`ava_network` TLS configs against avalanchego's real `crypto/tls` config in a
minimal two-process harness that captures *both* sides' handshake outcomes.

Scope boundary (chosen): **diagnose, then fix if small.** If the root cause is
localized and the fix is small + verifiable against the same harness in-sandbox
(e.g. a cert-template or rustls-config tweak), continue into the fix this
session under its own short design/plan. If it is a deep rustls↔Go protocol
issue, stop at the documented root cause + ordered next steps; the live arm
stays `#[cfg(feature="live")] #[ignore]` (red-under-live, not weakened).

## 3. Strategy: isolated minimal TLS repro (chosen)

Reuses the established go-oracle harness precedent — the
`tests/differential/go-oracle/` dir already holds standalone Go `main.go`
programs (`rust_plugin_handshake/`, `rust_plugin_lifecycle/`) that import
avalanchego and interop with the Rust side, built and run from `~/avalanchego`.

Rejected (per the brainstorm):

- **Instrument the full live path** — wire native `tracing`→LogFactory, set
  `SSLKEYLOGFILE` on both nodes, pcap+Wireshark. Tests the real node path but
  inherits the logging-wiring fix as a prerequisite and is far heavier to
  iterate. Kept as the heavy fallback for evidence capture (§6), not the venue.
- **Cert-structure differential only** — feed certs through
  `staking.ParseCertificate`/`crypto/x509` in unit tests with no live TLS. Useful
  but cannot observe the *handshake* stall (TLS framing / `CertificateVerify`
  signature / `tokio-rustls` timing) — folded into the harness's decisive cell
  (#2 toggles `VerifyConnection`) instead of standing alone.

## 4. Architecture — a two-process matrix harness

### Go side — `tests/differential/go-oracle/tls_handshake/main.go`

A standalone Go program, built and run from `~/avalanchego` (so it can import
the avalanchego `staking` / `network/peer` packages), driven by flags:

- `--role=server|client`
- `--addr=127.0.0.1:<port>`
- `--verify=staking|noop` — `VerifyConnection` is either avalanchego's real
  leaf-key policy (`staking.ValidateCertificate` over the parsed peer cert) or a
  no-op that accepts any peer cert.
- `--keytype=rsa|ecdsa` — which staking key type this Go process presents
  (Go default is RSA; ECDSA matches the Rust side).

It builds a `tls.Config` mirroring avalanchego's `peer.TLSConfig`: TLS1.3-only
(`MinVersion=MaxVersion=VersionTLS13`), `ClientAuth=RequireAnyClientCert`,
`InsecureSkipVerify=true`, the chosen `VerifyConnection`, no ALPN. In `server`
mode it accepts one conn and calls `tlsConn.Handshake()`; in `client` mode it
dials and calls it. Either way it prints a single JSON line to stdout:

- on failure: `{"ok":false,"error":"<verbatim Handshake() error>"}` — **the
  currently-missing server-side signal**;
- on success: `{"ok":true,"version":...,"cipher_suite":...,"peer_cert":{...}}`
  from `tlsConn.ConnectionState()`.

The Go cert is generated via avalanchego's own staking cert generator
(`staking.NewTLSCert()` / the ECDSA variant) so the harness uses Go's *real*
cert template, not a hand-rolled one.

### Rust side — gated live test in `crates/ava-network`

A `#[cfg(feature="live")] #[ignore]` test (gated like the other live arms;
early-returns without `$AVALANCHEGO_PATH`), placed in `ava-network` (closest to
the TLS code). It:

1. builds (once) the Go `tls_handshake` binary in `~/avalanchego`;
2. for each matrix cell, spawns the Go process in the appropriate role, then
   drives `Upgrader::client(client_config(&Identity::generate()))` or
   `Upgrader::server(server_config(&id))` against it;
3. records the exact `Error::Tls(..)` string (or success + derived peer NodeID)
   and the Go process's JSON line, and asserts the expected per-cell outcome
   (see §5).

## 5. The bisection matrix

Each cell localizes the fault. The decisive cell is **#2**.

| # | TLS client | TLS server | verify | Go key | Expected / purpose |
|---|-----------|-----------|--------|--------|--------------------|
| 1 | Rust | Go | staking | RSA | **The live failure.** Reproduce; capture *both* sides' errors. |
| 2 | Rust | Go | **noop** | RSA | Isolate cert-policy from pure TLS. **Pass here + fail in #1 ⇒ Go's `ValidateCertificate`/`ParseCertificate` rejects our cert.** Fail here too ⇒ pure TLS crypto/framing. |
| 3 | Go | Rust | (Rust `AvaClientCertVerifier`) | RSA | Reverse direction — is the stall direction-specific? |
| 4 | Rust | Rust | n/a | — | Control: each stack internally completes (proves harness soundness). Already covered by existing in-process upgrader tests; included as a sanity assertion. |
| 5 | Rust | Go | staking | **ECDSA** | Does the Go server's own key type change the outcome? |

## 6. Evidence capture

- **rustls trace:** `RUST_LOG=rustls=trace` — the `log`→`tracing` bridge already
  lands rustls handshake records (confirmed 2026-06-23). Shows the last
  client-side rung.
- **Go error:** the verbatim `Handshake()` error + `ConnectionState` JSON
  (§4) — the missing server-side signal.
- **Heavy fallback** (only if the above errors are opaque, e.g. a bare EOF):
  add a **diagnostic-only** `key_log = Arc::new(rustls::KeyLogFile::new())` to
  the harness client config (reads `SSLKEYLOGFILE`; **not** wired into the
  production `tls_config.rs`), plus `tcpdump` on loopback → Wireshark decrypt to
  read the last handshake message and any alert.

## 7. Decision tree → "fix if small"

- **#2 passes, #1 fails** → Go's `staking.ParseCertificate` / `ValidateCertificate`
  rejects our ECDSA-P256 cert. Next: `asn1`-dump the Rust cert DER vs a
  Go-generated staking cert; prime suspects are `SerialNumber=0` encoding, the
  validity window, or the signature algorithm (`ava-crypto/src/staking/tls.rs:55`
  and around). Fix = a cert-template tweak in `tls.rs`, re-verified against cell
  #1 in-sandbox. **Small → proceed to fix this session** under a follow-up
  design/plan; success = cell #1 flips to handshake-complete.
- **#1 and #2 both fail** → pure TLS interop (client-cert signature-scheme
  selection, or a `tokio-rustls` half-close / read-half timing). Localize via
  keylog/pcap (§6). May be a small config tweak or a deep rustls-behavior issue
  — decide at that point; if deep, stop at documented root cause + next steps.
- **#3 differs from #1** → asymmetry pinpoints which side's cert/role is at
  fault (e.g. our `AvaClientCertVerifier` vs Go's, or our server cert vs our
  client cert).

## 8. Risks & deferrals

- **Go toolchain gotcha** (recorded in memory `m9.15-live-mixed-net-blocker`):
  building the Go harness from `~/avalanchego` may hit the `mise 1.25.10` vs
  `nix 1.26.3` `GOROOT` mismatch. Build via the same path `~/avalanchego/scripts/build.sh`
  uses; the harness `go build` runs in that checkout's module context.
- **Diagnostic-only keylog:** the `KeyLogFile` hook is added *only* to the
  harness client config, never to production `tls_config.rs` (leaking session
  secrets in a real node is unacceptable). Enforced by keeping it in the test,
  not the library.
- **Pre-gate binary check:** before the matrix run, `./scripts/check_oracle_binary.sh`
  must print `OK` (binary commit == `~/avalanchego` HEAD, `rpcchainvm=45`), per
  CLAUDE.md — a stale binary would mean comparing against the wrong Go source.
- **No production TLS change in this spec:** any fix is gated behind a separate
  follow-up design; this spec delivers the harness + the recorded root cause.

## 9. Definition of done

1. Harness committed: `tests/differential/go-oracle/tls_handshake/main.go` + the
   gated `ava-network` live test. Builds clean; `lint-all` green; the gated arm
   early-returns without `$AVALANCHEGO_PATH` so per-PR CI is unaffected.
2. The matrix run in-sandbox (Go binary buildable at `~/avalanchego`) produces a
   captured **root-cause finding**: the exact failing rung + verbatim Go and
   rustls errors + which matrix cell isolates it, recorded as an AS-BUILT note in
   `plan/M9-interop-hardening.md` (M9.15).
3. If the root cause is localized and the fix is small + harness-verifiable →
   land the fix (cell #1 flips to handshake-complete) under a follow-up
   design/plan. Otherwise → documented root cause + ordered next steps; the live
   `mixed_network` arm stays `#[cfg(feature="live")] #[ignore]`.
