# M9.15 — Accept avalanchego v1/RSA staking certs in the rustls TLS-1.3 handshake

**Date:** 2026-06-25
**Milestone:** M9 (plugin interop + hardening) — live Go-interop blocker
**Scope:** verifier fix only (RSA key-loading and the live `mixed_network` arm are out of scope this round)

## Problem

A Rust `avalanchers` node cannot complete a TLS-1.3 mutual handshake with a Go
`avalanchego` peer that presents an **RSA** staking certificate. The handshake
stalls below the Avalanche application layer; the peer never establishes and
bootstrap never starts.

Root cause is pinned (recorded in `plan/M9-interop-hardening.md`, the
2026-06-25 repro-matrix AS-BUILT note, and
`docs/superpowers/specs/2026-06-24-m9.15-tls-handshake-repro-design.md`):

- avalanchego mints staking key pairs as **self-signed X.509 v1** certs. For RSA
  this is confirmed via `openssl x509 -in staking/local/staker1.crt`
  (`Version: 1 (0x0)`, `rsaEncryption`).
- Our custom verifiers `AvaServerCertVerifier` / `AvaClientCertVerifier`
  (`crates/ava-network/src/peer/verifier.rs`) already **accept** v1/RSA leaf keys
  in their `verify_server_cert` / `verify_client_cert` callbacks — those use our
  own `x509-parser`-based leaf-key policy (`validate_leaf_public_key`), which is
  version-agnostic and RSA-aware.
- But both verifiers' `verify_tls13_signature` impls delegate to
  `rustls::crypto::verify_tls13_signature(message, cert, dss, &self.supported)`,
  which builds a `webpki::EndEntityCert` from the leaf DER to extract the public
  key. **webpki enforces X.509 v3 and rejects v1 certs with
  `UnsupportedCertVersion`.** The repro matrix's CELL2 (`verify=noop`,
  `keytype=rsa`) isolates this exactly: the Rust client rejects the Go server's
  RSA cert with `Other(OtherError(UnsupportedCertVersion))`.
- ECDSA certs minted by current `staking.NewCertAndKeyBytes()` are v3, so the
  ECDSA cell (CELL5) already completes a full mutual handshake. The fault is
  specific to the v1 path that RSA certs take.

## Approaches considered

1. **Raw-key verification path (chosen).** rustls 0.23 exposes
   `rustls::crypto::verify_tls13_signature_with_raw_key`, which takes a
   `SubjectPublicKeyInfoDer` instead of a full certificate — no webpki cert
   parse, hence no v3/version gate. We already parse the leaf with `x509-parser`
   (v1-tolerant); extract the SPKI DER and feed it in. No new dependencies, and
   it reuses rustls's own `SignatureScheme`→algorithm dispatch.
2. **Manual signature verification via `ring`.** Extract the SPKI, map the
   `SignatureScheme` to a `ring` algorithm ourselves, and verify directly.
   Duplicates rustls's scheme→algorithm mapping and adds error surface.
   *Rejected.*
3. **Patch or fork webpki to allow v1 certs.** Heavy and off-policy (`00` §4 — no
   gratuitous extra crates / forks). *Rejected.*

## Design

All changes are localized to `crates/ava-network/src/peer/verifier.rs`.

- Add a private helper:
  `fn leaf_spki_der(cert: &CertificateDer<'_>) -> Result<SubjectPublicKeyInfoDer<'static>, rustls::Error>`.
  It parses the leaf with `x509-parser` and returns the full SubjectPublicKeyInfo
  DER (`cert.public_key().raw`) as an owned `SubjectPublicKeyInfoDer`. A parse
  failure maps to `rustls::Error::InvalidCertificate(CertificateError::BadEncoding)`,
  consistent with the existing `policy_to_rustls` mapping for unparsable certs.
- Both `verify_tls13_signature` impls (client-side and server-side) extract the
  SPKI via the helper and call
  `verify_tls13_signature_with_raw_key(message, &spki, dss, &self.supported)`
  instead of the cert-based `verify_tls13_signature`.
- Consequence: **all** TLS-1.3 handshakes (ECDSA-v3 and RSA-v1 alike) now verify
  the CertificateVerify signature through the raw-key path. This is uniform and
  intentional. The leaf-key *policy* enforcement is unchanged: it still happens
  in `verify_server_cert` / `verify_client_cert` (ECDSA must be P-256; RSA must
  be well-formed — modulus 2048/4096 bits, positive, odd; exponent 65537). So
  the security posture (peer authenticated by its self-signed leaf public key,
  matching Go's `InsecureSkipVerify + VerifyConnection(ValidateCertificate)`) is
  preserved — we have only removed webpki's spurious cert-version gate from the
  signature step.

`verify_tls12_signature` is unchanged (TLS 1.2 is disabled at config level and
those impls already return `Tls12NotOffered`).

## Testing (TDD — failing test first)

- **Reproduction test (the load-bearing proof):** an in-process rustls
  server↔client TLS-1.3 handshake where the server presents an **X.509 v1 RSA**
  cert and the client uses `AvaServerCertVerifier`. Before the fix this fails
  with `UnsupportedCertVersion`; after the fix the handshake completes. Mirror
  the same with a v1 RSA *client* cert verified by `AvaClientCertVerifier`. This
  is the in-process equivalent of repro CELL2.
- **Fixture:** commit a static RSA test cert + matching PKCS#8 private key,
  2048-bit modulus, exponent 65537, **no v3 extensions** (so it serializes as
  X.509 v1) — shape-matching avalanchego's staking cert. Generated once via
  `openssl` and documented in-comment with the exact generation command. The
  matching private key is required so the test peer can sign CertificateVerify.
  `rcgen` emits v3 certs and cannot produce the v1 shape, so a committed fixture
  is the faithful way to reproduce the failure.
- **Regression:** the existing ECDSA-backed handshake tests
  (`crates/avalanchers/tests/two_node_convergence.rs`,
  `crates/avalanchers/tests/networked_bootstrap.rs`, and any in-crate
  `ava-network` handshake tests) must stay green — proving the uniform raw-key
  path did not regress the common ECDSA case.

## Risks / unknowns

- `verify_tls13_signature_with_raw_key` must be reachable with the workspace
  rustls feature set (`ring, std, tls12`). The repro confirmed the symbol is
  exported from `rustls::crypto`; if a feature flag is nonetheless required, that
  is a one-line `Cargo.toml` change.
- `x509-parser` 0.16's `public_key().raw` must be the complete SubjectPublicKeyInfo
  DER that `verify_..._with_raw_key` expects. The reproduction test pins this end
  to end (a mismatch would fail signature verification, not just parsing).
- webpki had exactly one touchpoint in our verifiers — `verify_tls13_signature`.
  The cert-validation callbacks are already webpki-free. So removing that one
  touchpoint is expected to fully close the v1 gap; the reproduction test
  confirms it rather than assuming it.

## Out of scope (deferred)

- **RSA staking-key loading** (`Identity::from_pem` accepting RSA PKCS#1 so a
  Rust node can *present* a Go-minted RSA staker cert) — interop-gap #1. Not
  needed for interop: the Rust follower presents an ECDSA cert; this fix is about
  *accepting* an RSA peer's cert.
- **Flipping the live `mixed_network` two-binary arm green.** It stays
  `#[cfg(feature="live")] #[ignore]`. This round's proof is the in-process
  v1-RSA handshake test. The live arm is the documented follow-up.
- The release-build on-disk DB gap (interop-gap #2) is unrelated to TLS and out
