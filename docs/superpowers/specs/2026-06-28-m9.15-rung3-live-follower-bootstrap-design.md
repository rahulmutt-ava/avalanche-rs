# M9.15 rung-3 — live Rust-follower bootstrap against a Go cluster

**Date:** 2026-06-28
**Status:** Design (approved for plan)
**Area:** `crates/ava-network` (dial/connect instrumentation), `crates/avalanchers/src/wiring/chains.rs` (gate/start/frontier instrumentation), `tests/differential` (single-Go-beacon repro harness + live arms)
**Predecessor:** `2026-06-24-m9.15-production-network-consensus-wiring-design.md` (production wiring, merged); `plan/M9-interop-hardening.md` Task M9.15 AS-BUILT (2026-06-28, Stage 1 GREEN / Stage 2 RED)

## 1. Problem

The live `differential::mixed_network` arm brings up 5 Go genesis validators
(full mesh, BLS-signed quorum) plus one Rust ECDSA follower configured to
bootstrap from all 5. As of 2026-06-28:

- **Stage 1 GREEN** — the 5 Go validators reach quorum and bootstrap P/X/C.
- **Stage 2 RED** — the Rust follower completes TLS-1.3 mutual handshake and the
  app-layer Handshake (it replied a `PeerList` to one Go validator, staker4),
  then **goes silent**: no `GetAcceptedFrontier`/`AcceptedFrontier`/`Ancestors`
  is ever emitted, and `isBootstrapped(P/X/C)` never flips true within the
  180 s harness timeout (`tests/differential/tests/mixed_network.rs`).

The mechanism is understood from the production wiring code. The bootstrapper's
`handler.start()` — and therefore the `GetAcceptedFrontier` broadcast — is
deferred behind the beacon-connectivity gate
(`crates/avalanchers/src/wiring/chains.rs:1336-1346`):

```rust
if gate.wait_for(|&v| v).await.is_ok() {
    let inner = handler.start();
    let _ = inner.await;
}
```

The gate fires when `BeaconManager` counts
`required_conns = (3·num_beacons + 3) / 4` connected beacons
(`crates/ava-node/src/init/networking.rs:399-415`). With 5 beacons that is
**4**. The live run formed only **1** beacon connection, so `wait_for` blocks
forever, `handler.start()` never runs, and no bootstrap traffic is emitted. The
observed "connect then silence" is fully explained by the connectivity gate
never reaching threshold.

The gate, count, and downstream engine path are **not** suspected: the
Rust↔Rust equivalent (`crates/avalanchers/tests/two_node_convergence.rs`) drives
a follower `Node` to `NormalOp` through the same production path, and
`prod_loop_close.rs` proves inbound→engine→outbound across the production seam.
The unexplained fact is purely: **why did only 1 of 5 beacon connections form**,
when all 5 are the same Go binary and one of them (staker4) accepted the
follower.

## 2. Goal

Identify and fix the root cause that prevents the live Rust follower from
climbing past the connect→bootstrap-start boundary against a real Go cluster, so
that `differential::mixed_network` Stage 2 reaches `NormalOp`
(`isBootstrapped(P/X/C)=true`) — or, if a deeper rung is exposed, isolate and
record it honestly as the next gated rung.

This is a **systematic-debugging** task with a single live data point. The plan
is a live diagnosis loop, not a speculative fix. The only pre-committed code is
diagnosis infrastructure (a bisection harness + env-gated instrumentation); the
fix is scoped by live evidence.

## 3. Approach: diagnose by bisection

### 3.1 The bisection probe — a single-Go-beacon follower

Add a cheaper live arm: one Go validator + one Rust follower that bootstraps
from that single node. A 1-node Go cluster self-bootstraps trivially, and the
follower's gate threshold collapses to `required_conns = (3·1 + 3) / 4 = 1`. A
single beacon connection therefore fires the gate. This bisects the two possible
failure halves:

- **If the follower bootstraps from 1 Go node** → the entire
  gate→start→frontier-out→frontier-in→NormalOp path works against real Go, and
  the 5-validator failure is **purely connectivity** (forming 4 of 5
  connections). Scope collapses to the dial/handshake layer (rungs 1-4 below).
- **If the follower also stalls with 1 Go node** → the engine/bootstrap-start
  path itself breaks against a real Go peer (frontier query mis-addressed, or
  inbound `AcceptedFrontier`/`Ancestors` not routed back to the engine). The fix
  is at rungs 6-9, a different and deeper change.

The single-beacon arm is also permanently cheaper to run than the 5-validator
arm, so it becomes the fast live-iteration target.

### 3.2 The rung ladder

The follower must climb this ladder; instrumentation (§4.2) marks each rung so a
live run pinpoints the break:

```
1. dial          follower dials all N beacons (dialer loop, tracked_ips)   [net_impl]
2. TLS upgrade   mutual TLS-1.3 completes per beacon                       [proven live, e06f0a0]
3. app handshake Handshake + knownPeers bloom accepted per beacon         [proven for 1 (staker4)]
4. connected     BeaconManager.connected counts each beacon (conns→N)     [needs >= required_conns]
5. gate fires    on_sufficiently_connected = true                         [blocked live: 1 < 4]
6. start()       handler.start() runs -> engine activates                 [gated on rung 5]
7. frontier out  GetAcceptedFrontier broadcast via OutboundSender         [never reached live]
8. frontier in   AcceptedFrontier/Ancestors routed RouterBridge->engine   [never reached live]
9. NormalOp      bootstrap Finished -> isBootstrapped(P/X/C)=true
```

Live evidence places the 5-validator break at **rung 4**. The single-beacon
bisect drops `required_conns` to 1, firing rung 5 on the one proven connection
and thereby exercising rungs 6-9 against real Go for the first time.

## 4. Components

### 4.1 Single-Go-beacon repro harness — `tests/differential/`

- `tests/differential/src/network.rs`: `boot_single_go_beacon(seed)`, a sibling
  of `boot_mixed`. Spawn one Go validator (reuse `spawn_role_node`, `NodeLaunch`,
  `local_staker(1)` + `local_signer_key(1)`); wait for it to self-bootstrap
  (`await_bootstrapped(beacon_api, &["P","X","C"], …)`); then spawn the Rust
  follower (fresh ECDSA staker, `bootstrap = [that one Go node]`,
  `signer_key_file: None`) and `await_bootstrapped` on the follower.
- `tests/differential/tests/mixed_network.rs`: new
  `#[cfg(feature="live")] #[ignore]` test `mixed_network_single_beacon`.
- Reuses the existing `check_oracle_binary.sh` pre-gate and `log_tail` reporting.

### 4.2 Env-gated rung instrumentation

Targeted `tracing` at each rung, at `debug`/`trace` level so it is inert in
normal runs and surfaced with a raised `RUST_LOG`:

- `crates/ava-network/src/network/net_impl.rs`: per-beacon dial outcome in
  `handle_dial` (TCP connect / TLS upgrade / spawn) and handshake-complete vs
  rejected in `watch_peer` — each with NodeId and the error on failure. The
  existing swallowed/`debug` dial-failure log is extended, not duplicated.
- `crates/ava-node/src/init/networking.rs`: `BeaconManager::connected` logs
  `conns/required_conns` on each increment and on gate fire.
- `crates/avalanchers/src/wiring/chains.rs`: gate-fired + `handler.start()` entry
  (around line 1342); first `GetAcceptedFrontier` emission at the
  `OutboundSender` seam; inbound `AcceptedFrontier`/`Ancestors` routed at the
  `RouterBridge` seam.

Style follows the existing T6 handshake rung logging; no new logging framework.

### 4.3 Live run procedure (documented loop)

`check_oracle_binary.sh` (rebuild ~/avalanchego on FAIL) → build `avalanchers` →
run the single-beacon arm with `RUST_LOG` raised → read `log_tail` / follower
log → identify the broken rung → (fix) → re-run. Outcomes recorded in
`plan/M9-interop-hardening.md` as AS-BUILT notes (the established pattern).

### 4.4 The fix (scoped after the bisect)

Held open deliberately — chosen by live evidence, TDD'd where a unit/integration
test can express it offline. Likely candidates:

- **If connectivity (rungs 1-4) is the broken half:** outbound dial/handshake to
  non-first beacons (e.g. a race or early-return in the dialer/`handle_dial`
  path), knownPeers/PeerList re-dial, or a per-beacon signed-IP/timing rejection
  that staker4 happened to pass.
- **If the engine (rungs 6-9) is the broken half:** `GetAcceptedFrontier`
  beacon-set addressing in the `OutboundSender`, or inbound bootstrap-reply
  routing through `RouterBridge` → shared `ChainRouter` → engine.

## 5. Definition of done

- **Bisect arm result captured:** `mixed_network_single_beacon` either reaches
  `isBootstrapped(P/X/C)=true` from one live Go node, or its broken rung is
  pinpointed by instrumentation and recorded.
- **Root cause identified from evidence**, not guessed; the fix is TDD'd where an
  offline test can express it (e.g. dialer-redials-all-tracked, or inbound
  frontier-reply routing).
- **5-validator arm re-run live:** `differential::mixed_network` Stage 2 reaches
  `NormalOp`, or the next isolated rung is recorded honestly and stays gated.
- **No offline regressions:** `mixed_network_replay_is_deterministic` + proptest
  untouched and green; `lint-all` + `test-unit` green; live arms stay
  `#[cfg(feature="live")] #[ignore]`.
- **AS-BUILT recorded** in `plan/M9-interop-hardening.md`.

## 6. Risks & notes

- **Non-determinism of "1 of 5":** if the connection count varies run-to-run, the
  bug is a dialing/handshake race; per-beacon outcome logging (§4.2) is required
  to catch it rather than a single aggregate count.
- **Gate-wait cancellation caveat (known, inert here):** the gate-wait task
  (`chains.rs:1336`) only unwinds when the watch sender drops. We *want* it to
  fire, not cancel, so this is not a concern for this work; noted for completeness.
- **Live cost:** the 5-validator run is heavy (Stage 1 alone needs quorum); the
  single-beacon arm is the fast iteration target and the primary DoD probe.
- **Fix may be ava-network-shaped, not engine-shaped:** if Go rejects the
  follower for a non-validator or signed-IP reason, the change lands in the
  ava-network handshake path, not the consensus engine. Scope stays bounded to
  whichever rung the evidence breaks at; SAE dispatch and other M9 deferrals are
  out of scope.
