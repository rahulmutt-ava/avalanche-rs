## Background — where we are

The M9.15 live `mixed_network` blocker chain has been climbing rungs, each fixed in turn:

1. **TLS-1.3 mutual handshake** (rustls ↔ Go) — rustls rejected Go's RSA v1 staking cert. Fixed (`e06f0a0`), confirmed live (`2ad33d2`).
2. **`knownPeers.filter` bloom** — Go rejected the Rust `Handshake` with `invalid num hashes`. Fixed by porting the write-side `x/bloom` + 40-byte `gossip_id` (`31a0cb0`…`d607804`).

**Confirmed live 2026-06-28 (this session):** running the gated arm against `~/avalanchego@cbea62895c` (rebuilt `avalanchers` release including the bloom fix), **both nodes report `info.peers numPeers:1`** — Go no longer logs the malformed-message rejection. The bloom fix works in production; `Peers:1` is real.

**The next rung (rung 2) is a harness topology bug, not a Rust networking bug.** The live arm boots a single Go validator (`staker1`) against the stock `genesis_local.json`, which hard-codes **5 initial stakers** (`staker1..5`; the first is exactly our beacon, `NodeID-7Xhw2mDxuDS44j42TCB6U5579esbSt3Lg`). With 1 of 5 validators (20 % of stake) connected, the Go beacon never reaches bootstrap quorum — its log repeats `peer/peer.go:784 failed to get peer uptime … error:"still bootstrapping"` indefinitely. A node that is itself still bootstrapping cannot serve an accepted frontier, so the Rust follower's `await_bootstrapped` times out at 180 s and the arm fails (honestly).

## Goal

Make the gated `mixed_network` live arm boot a topology that **actually bootstraps**, so it exercises the real interop guarantee — *Rust follows a live Go consensus network* — and empirically exposes whether a further rung (Rust↔Go bootstrap-protocol parity) exists.

Non-goals: changing production node/network code; un-gating the arm into CI; altering consensus semantics (no sybil-protection shortcuts); hand-building a custom genesis.

## Approach (chosen: Option A — full 5-validator Go quorum)

Boot **all 5 stock-genesis Go validators** (`staker1..5`, RSA — Go loads them natively) so the primary network reaches quorum and bootstraps, then have the **Rust ECDSA follower** (fresh cert, 0 weight, non-validating) join and follow. No genesis surgery, no semantic shortcuts, highest fidelity. Cost: 6 node processes.

Rejected alternatives:
- **B (custom 1-validator genesis via `--genesis-file-content`)** — 2 procs, but requires hand-building the P-Chain + C-Chain/coreth genesis (error-prone); deferred.
- **C (`--sybil-protection-enabled=false`)** — 2 procs, lightest, but alters consensus semantics and Go/Rust flag parity is untested; rejected for an acceptance arm.

## Components changed

All in `tests/differential/`:

### 1. Multi-bootstrapper support (`src/livenet.rs`)
- `Bootstrap` becomes a *list* of `(ip, id)` peers (e.g. `Vec<BootstrapPeer>`), or `NodeLaunch.bootstrap` carries `Vec<BootstrapPeer>`.
- `node_args` emits comma-joined `--bootstrap-ips=ip1,ip2,…` and `--bootstrap-ids=id1,id2,…` (avalanchego accepts comma-separated lists). Empty list ⇒ no flags (beacon/seed behavior preserved).
- The existing single-bootstrap unit tests (`no bootstrap-ips on beacon`, etc.) update to the list shape; add a multi-peer arg-rendering test.

### 2. `boot_mixed` → 5 Go validators + 1 Rust follower (`src/network.rs`)
- Allocate **12 ports up front** via `free_ports(12)`: 5 Go × (http, staking) + 1 Rust × (http, staking).
- Derive the 5 Go NodeIDs deterministically. They are fixed for the `local` network and pinned in the embedded `genesis_local.json` `initialStakers` (order = `staker1..5`). Read them from genesis (or a small vendored constant table) so bootstrappers can be wired **before** spawning — avoiding a scrape-before-wire ordering dependency.
- Spawn 5 Go nodes; node *i* uses `stakerI.crt/key` from `$AVALANCHEGO_SRC/staking/local/`. Each Go node is configured with the **other four** Go `(ip, id)` pairs as bootstrappers (full mesh ⇒ guaranteed quorum).
- **Wait for the Go cluster to bootstrap** — poll `info.isBootstrapped` for P/X/C on `staker1` until true (generous deadline). This is the checkpoint proving the 5-validator quorum formed; it cleanly separates rung 2 from rung 3.
- Then spawn the Rust ECDSA follower (`generate_staker`, 0 weight) bootstrapping from the Go set, and run the existing `await_bootstrapped` wait for the Rust node.
- The oracle pre-gate (`check_oracle_binary.sh`) and `kill_on_drop` teardown are unchanged.

### 3. `Network` accessors (`src/network.rs`)
- With 6 nodes, fixed positional indices break. Add `go_beacon()` (returns `staker1`, `nodes()[0]`) and `rust_follower()` (returns the last node) accessors.
- The `mixed_network.rs` live test uses these instead of `nodes().first()` / `nodes().get(1)`.

### 4. Live test (`tests/mixed_network.rs`)
- The `#[cfg(feature="live")]` `mixed_network` test switches to `net.go_beacon().api_base` / `net.rust_follower().api_base`. Its existing flow (record C height → `drive_c_transfer` on the Go beacon → `await_same_c_height` → normalized `Observation` no-fork assertion) is otherwise unchanged.
- **Offline arms are untouched** (`mixed_network_replay_is_deterministic` + proptest stay exactly as-is — pure in-process, no node spawn).

## Staged validation (deliverable vs discovery)

- **Stage 1 (the deliverable):** the 5-node Go cluster reaches `isBootstrapped` P/X/C. Confirms the topology fix. If this does *not* happen, the quorum / bootstrapper wiring is itself wrong — a harness bug to fix before anything else.
- **Stage 2 (discovery):** observe whether the Rust follower then bootstraps.
  - **Yes** → drive the C-transfer + no-fork assertion; the arm goes green; M9.15 live is functionally complete (modulo nightly cadence).
  - **No** → we have cleanly isolated **rung 3** (Rust↔Go bootstrap-protocol parity — `GetAcceptedFrontier` / `Ancestors` exchange). The epoch-0 `lastReceived` on the Rust peer hints Rust may not process inbound app messages from Go, but this is untested until Stage 1 passes. Record the finding honestly; the arm stays `#[cfg(feature="live")] #[ignore]`; rung 3 becomes the next investigation (separate spec/plan).

## Risks & caveats

- **Resource:** 5 Go nodes each run P+X+C (coreth EVM is memory-heavy) + 1 Rust = 6 processes. Should be fine on a dev box; if memory bites, a later optimization can trim to the minimum quorum (e.g. 3 of 5). Start with all 5 for correctness.
- **Sandbox env (carried forward):** restore `~/avalanchego` to the validated pin (`git checkout cbea62895c`) if auto-pull drifted HEAD; rebuild `avalanchers` release when commits postdate the binary; set `TMPDIR` to the scratchpad (system temp is not sandbox-granted — `generate_staker` write fails `Permission denied` otherwise); run the live arm with `cargo test` (not nextest — 180 s boot exceeds nextest's 120 s per-test timeout).
- **Rung 3 unknown:** Stage 2 may reveal the arm still cannot reach green; that is an acceptable, expected outcome of this work (it converts a topology unknown into a precise protocol question).
- **No CI impact:** the arm remains gated; offline arms unchanged.

## Definition of done

1. `boot_mixed` brings up 5 Go validators + 1 Rust follower with full-mesh bootstrappers; the Go cluster reaches `isBootstrapped` (Stage 1 passes when run live).
2. Multi-bootstrapper `node_args` rendering + `Network` accessors land with updated unit tests; offline arms still green; `cargo build`/clippy/fmt clean.
3. The live arm runs cleanly through Stage 1; Stage 2's outcome (Rust bootstraps, or rung-3 isolated) is recorded in `plan/M9-interop-hardening.md` + memory.
4. Arm stays `#[cfg(feature="live")] #[ignore]`.
